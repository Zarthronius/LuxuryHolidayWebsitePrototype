Security Considerations

In the interests of security on this website, there are a few considerations. The Open Web Application Security Project (OWASP) Foundation (2017) have identified the top 10 web application risks. From these 10, one of the most appropriate risks is SQL injection (particularly on the admin submit holiday form). SQL injection allows users to manipulate database tables and records associated with the website either directly by sending SQL queries or perhaps through second-order queries. Second-order queries, in particular, allow the user to insert malicious SQL code within the database. These are then later called to be run by the DBMS through a second query. Another potential issue is security misconfiguration, potentially through misconfigured HTTP headers. These allow users to see sensitive information, which could be exploited (such as the server system) and lets attackers use known exploits easier. Another potential threat identified by Malwarebytes (2017) is homogram (also known as homoglyph) attacks. Homograms are when a malicious party, or "threat actor", will create fake domains with a name almost identical to those of another target domain. However, these false domains will use characters that are indistinguishable from those in the original domains name. The purpose of this is to trick users into opening the malicious site. An example of how this commonly occurs is through the use of Cyrillic letters.

With all these potential threats in mind a simple, yet effective way to help resolve this would be ensuring all systems are running the latest software patches at all times. Doing so helps ensure that new security patches and optimisations get implemented, making it more difficult for any malicious threat actors to cause damage to the website. In the same vein as this, other key considerations would be limiting HTTP header details to reduce the amount of exploitable information provided to regular users (e.g. which server system is in use). By also protecting the site with HTTPS through certification, giving end to end encryption between users and the server, and as Kayce Basques, web.dev (2020) states, thus preventing third parties from intercepting data packages and passively listening to user website interactions, and provides an additional layer of anonymity to the user. One other essential consideration would be creating prepared statements to sanitise and entries to the form on the admin page, and perhaps even a secure login page with hashed passwords. Doing so allows only essential admin users access to the database. And lastly, the use of prepared statements would also drastically reduce any risks of SQL injection attacks to records saved within the DBMS.
